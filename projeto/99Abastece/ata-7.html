3. Pontos de Discussão e Decisões (Validação com Victoria)

  • Risco da Atualização via Python (Máquina da Roberta):
    • Victoria: Concorda que é um risco e que a solução ideal não é depender da máquina da Roberta. Prioridade é buscar uma solução mais robusta (nuvem ou VM interna, a ser discutido com Isaac/TI 99).
    • Decisão: Manter o processo atual via Python API como plano A para o Go-Live, mas a 99 buscará internamente uma alternativa mais estável em paralelo. A C3C deve garantir que a API seja flexível para apontar para a nova origem no futuro.
  • Força-Tarefa:
    • Victoria: Concorda com a necessidade de manter a possibilidade de ajudar um analista que está com dificuldades, mas prefere o modelo de transferência de contatos não trabalhados ou em andamento pela supervisora (Thaís), em vez de um analista "roubar" a conversão do outro.
    • Decisão: A funcionalidade de transferência manual de oportunidades/conversas pela supervisora atende à necessidade. A meta permanece com quem recebeu o 1º retorno.
  • Metas Separadas ou Juntas (Churn vs. Never Used):
    • Victoria: Confirmou que as metas de conversão DEVEM ser separadas por perfil (Churn e Never Used), pois as taxas esperadas são diferentes.
    • Trabalho Híbrido (Analista trabalhar com ambos perfis):
      • Confirmou o desejo de ter essa flexibilidade futura (parametrizável).
      • Decisão Fase 1: No início, cada analista trabalhará com apenas um perfil por campanha/mês (ou Churn ou Never Used) para simplificar a adaptação. A C3C deve preparar a estrutura para permitir a configuração híbrida no futuro (ex: via tipo de registro ou campo na Oportunidade/Lead).
  • Reativar Contato Expirado (Repescagem):
    • Victoria: Concorda com a análise de complexidade vs. prazo. É um "nice to have", não essencial para a Fase 1.
    • Decisão: Repescagem fica fora do escopo da Fase 1.
  • Nova Estrutura de Metas (Pós-Automação):
    • Victoria/Roberta: Concordam que a meta de acionamento (80%) perderá o sentido. Precisam discutir internamente e testar novas métricas (Ex: SLA de tempo de resposta, nº de retornos obtidos, taxa de contato efetivo).
    • Decisão Fase 1: O cálculo de bonificação no Salesforce será focado inicialmente apenas na meta de conversão (com faixas parametrizáveis). A 99 definirá posteriormente como (ou se) irá incorporar novas métricas de esforço/qualidade no bônus.

4. Apresentação e Alinhamento Solução To-Be (C3C)

  • Estrutura Geral:
    • Carga: Manual (Base Mestra 1-2x/mês) + API Diária (Abastecimentos).
    • Objetos Principais: Conta (Motorista - usando existente), Oportunidade (Ciclo de contato/conversão), Tarefa/Evento (Registro de interações).
    • Campanha Salesforce: Usada para agrupar os motoristas do mês e gerenciar os disparos.
    • Distribuição: Regra de atribuição automática baseada em resposta e ociosidade do analista.
    • Carteira: Oportunidade associada ao analista que recebeu o 1º retorno.
    • Comunicação: Integração com WhatsApp (número único BAPI).
    • Follow-ups (Não Respondidos): Fluxo automático/agendado (ex: a cada 24h, até 5-7 tentativas), gerenciado centralmente.
    • Relatórios/Dashboards: Para acompanhamento de metas (Conversão), performance individual/equipe, status do funil.
  • Fluxo Detalhado (Validado):
    • Roberta carrega base mestra -> Cria/ativa Campanha do Mês.
    • Roberta/Thaís configura regras de disparo diário (volume, horário, template inicial, prioridade Churn).
    • Sistema dispara 1ª abordagem.
    • Motorista Responde -> Mensagem chega no Salesforce -> Sistema identifica analista ocioso -> Cria Oportunidade e associa ao analista -> Analista recebe notificação/tarefa.
    • Analista interage via Salesforce (interface WhatsApp).
    • Se Motorista NÃO Responde -> Sistema dispara 2ª abordagem (agendada) -> ... -> até limite de tentativas. Se responder em qualquer etapa, vai para o fluxo de resposta acima.
    • Roberta envia atualização de abastecimento via API -> Salesforce atualiza Oportunidade correspondente para "Ganha".
    • Se Oportunidade atinge 7 dias sem "Ganha" -> Sistema atualiza para "Perdida".
    • No fim do mês -> Fecha Campanha -> Roberta/Victoria extraem relatórios para cálculo de bônus (focado em conversão).
  • Pontos de Atenção Técnicos (Tarcísio):
    • Mapeamento correto do Driver ID.
    • Lógica robusta para distribuição por ociosidade e gestão de capacidade.
    • Tratamento de erros na API de abastecimento.
    • Configuração dos fluxos automáticos de follow-up e expiração de oportunidade.
